# 车辆轨迹自动提取算法说明（峰图 + 动态规划 + 轨迹拼接）

本文档总结当前 GUI/后台实际使用的轨迹提取算法，包括数学模型、物理意义、参数选择依据与调参建议。

![image01.png](image01.png)

## 1. 问题建模

- 输入：DAS 振动数据矩阵 `data[ch, t]`，采样率 `fs`，道间距 `dx_m`。
- 输出：多条车辆轨迹 `Track`，每条轨迹由一组真实峰点 `TrackPoint` 组成。
- 方向约束：由 GUI 手动指定 `forward` 或 `reverse`。
- 速度约束：由 GUI 指定 `[vmin, vmax]`（km/h）。

物理上，车辆在空间-时间图上表现为一条斜线（或轻微弯曲线）。

- 当速度恒定时：
  - `dt/dx ≈ 1/v`（时间-空间斜率）
- 当轻微加减速时：
  - 斜率缓慢变化，轨迹不再严格直线

  算法目标是在噪声和漏检下，找到“方向一致 + 速度可行 + 强峰优先”的峰点路径。

---

## 2. 总体流程

当前实现分为 7 步：

1. **模板增强**（多尺度高斯）
2. **候选峰提取**（`find_peaks`）
3. **速度约束建边**（跨道连接）
4. **动态规划求最优路径**（单条轨迹）
5. **边界轨迹放宽**（窗口起止附近）
6. **NMS 抑制并迭代提取多车**
7. **轨迹后拼接**（解决轻微加减速 + 漏点断裂）

整小时模式会分块提取后进行跨块合并与全局去重。

## 3. 数学原理

## 3.1 模板增强

先取绝对值：

`A(ch,t) = |data(ch,t)|`

对每个高斯模板尺度 `sigma_s` 做 1D 高斯平滑：

`R_sigma(ch,t) = G(sigma_s * fs) * A(ch,t)`

当前实现在增强阶段使用时间降采样（`enhance_decimate=2`），即先在 `t/2` 网格上做模板增强与峰检，再映射回原采样点。  
这样主耗时步骤计算量约减半，对秒级轨迹定位精度影响很小（本任务速度量级为数十秒/公里）。

取模板库最大响应：

`R(ch,t) = max_sigma R_sigma(ch,t)`

意义：增强“脉冲形”信号，抑制随机噪声；多尺度模板适配不同脉宽。

当前默认模板（秒）：`(0.06, 0.10, 0.14, 0.18)`。

---

## 3.2 候选峰与节点得分

在每道增强信号 `R(ch,:)` 上做峰值检测：

- `prominence >= P`
- 相邻峰间距 `>= D`

每个峰构成一个节点，节点得分：

`node_score = prominence + 0.2 * (amp / median_amp_channel)`

其中 `amp = |data(ch, t_peak)|`。

意义：

- `prominence` 表示峰相对局部背景的突显程度（抗基线漂移）
- `amp` 作为次级项，避免只看形状不看能量

---

## 3.3 速度窗约束（物理可行边）

设相邻候选点在 `dch` 个道上的间隔为 `Δx = dch * dx_m`，时间差为 `dt`。

速度窗（m/s）：

- `vmin = speed_min_kmh / 3.6`
- `vmax = speed_max_kmh / 3.6`

### forward 方向

`dt ∈ [Δx / vmax, Δx / vmin]`

### reverse 方向

`dt ∈ [-Δx / vmin, -Δx / vmax]`

只有满足该区间的候选点才允许连边。

---

## 3.4 动态规划路径优化

这一段是整套算法的核心。可以把它理解成：

“在每个候选峰点处，记录一条到这里为止的最优轨迹；再把这些局部最优逐道向前推进，最后得到全局最优。”±

### 3.4.1 状态定义（代码里实际存了什么）

对每个候选点 `(ch, idx)`，维护以下量：

- `dp_score[ch][idx]`：到该点为止的最优累计得分
- `dp_len[ch][idx]`：最优路径长度（经过的点数）
- `dp_speed[ch][idx]`：该路径的“当前速度估计”
- `dp_prev_ch[ch][idx], dp_prev_idx[ch][idx]`：回溯父节点
- `dp_tmin[ch][idx], dp_tmax[ch][idx]`：该路径覆盖的时间范围（用于边界放宽判定）

所以 DP 不只是“一个分数”，而是“分数 + 长度 + 速度 + 父指针 + 时间包络”的组合。

### 3.4.2 转移方程（从上一道/上几道连过来）

对当前点 `j`，在 `dch=1..max_skip_channels` 的前序道里找可连的前驱 `i`：

`DP(j) = max_i { DP(i) + score(j) - Pen_speed(i,j) - Pen_skip(i,j) }`

其中：

- `Pen_skip = lambda_skip * max(0, dch - 1)`
- `dch` 是跨越道数（`dch=1` 表示相邻道，`dch>1` 表示“跳道容错”）

直觉上：

- `score(j)` 鼓励走强峰
- `Pen_speed` 抑制“速度突变”
- `Pen_skip` 抑制“跳太远”

### 3.4.3 速度变化惩罚（为什么第三个点会偏，往往就在这里）

设连接 `i -> j` 产生瞬时速度 `v_ij`，上一段速度估计 `v_prev`：

`delta_v = |v_ij - v_prev|`

`excess = max(0, delta_v - v_tol)`

`Pen_speed = lambda_speed * (excess / (vmax - vmin))^p`

并截断：

`Pen_speed = min(Pen_speed, speed_penalty_cap)`

默认：

- `v_tol = 18 km/h`
- `p = 0.85`

含义：

- `delta_v <= v_tol`：基本不罚（允许小幅加减速）
- 超出容忍后再罚（避免过早断轨）
- `p < 1`：惩罚增长更平滑，减少“轻微偏差直接崩掉”的硬切换

### 3.4.4 速度估计如何更新

不是简单用 `v_ij` 覆盖，而是平滑更新：

`v_new = 0.7 * v_prev + 0.3 * v_ij`

这样做的目的：

- 降低单个异常点对后续路径的“牵引偏差”
- 让轨迹斜率变化更连续

### 3.4.5 数值例子（最容易建立直觉）

假设：

- `dx=100m`
- 速度窗 `60~120 km/h`（即 `16.67~33.33 m/s`）

若 `dch=1`（相邻道）：

- `dt` 合法范围约 `[100/33.33, 100/16.67] = [3s, 6s]`

若 `dch=2`（跳过 1 道）：

- `dt` 合法范围约 `[6s, 12s]`
- 会额外吃到 `Pen_skip`

所以“跳道不是禁止”，而是“允许但要付代价”。

### 3.4.6 平分时如何决策

代码里有 tie-break：若得分几乎相同，优先更长路径（`dp_len` 更大）。  
这能减少“短而高分噪声串”压过“长而稳定真轨迹”的情况。

### 3.4.7 一段伪代码（与实现一一对应）

```text
for ch in channels:
  for node j in channel ch:
    best = node_score(j)
    for dch in 1..max_skip_channels:
      pch = ch - dch
      for node i in channel pch:
        if speed_window_ok(i, j, dch):
          cand = dp(i) + score(j) - pen_speed(i,j) - pen_skip(dch)
          update_if_better(cand, with_tie_break_by_length)
    store dp_score / dp_len / dp_speed / parent / tmin_tmax
```

## 3.5 轨迹合法性判据

严格条件：

- `len(track) >= min_track_channels`
- `total_score >= min_track_score`

边界放宽（解决“刚进入/刚离开窗口”的短轨迹）：

当轨迹触及窗口边界时间带时，允许：

- `len(track) >= edge_min_track_channels`
- `score >= edge_min_score_scale * min_track_score`

当前默认：

- 边界时间带 `15s`
- `edge_min_track_channels = 6`
- `edge_min_score_scale = 0.55`

### 3.5.1 为什么边界放宽很关键

窗口边缘处的车，天然只会出现“半条轨迹”：

- 刚进入窗口：只看到后半段
- 即将离开窗口：只看到前半段

如果仍然用严格门限（例如必须 12 道），这些真实轨迹会被系统性漏掉。  
因此边界放宽本质上是“观测不完整时的公平判定”，不是“放水”。

### 3.5.2 严格判据与放宽判据的关系

真正被接受的集合是：

`Valid = StrictValid OR EdgeRelaxedValid`

也就是：

- 非边界区域：基本还是靠严格判据
- 边界区域：允许短轨迹但仍要有足够得分

### 3.5.3 什么时候会误放宽

如果 `edge_time_margin_seconds` 过大，会把大量中间区域也当边界，误检会上升。  
建议保持 `10~20s` 区间，通常 `15s` 比较平衡。

## 3.6 多车提取与 NMS

每次提取当前最优轨迹后，对其邻域候选点做抑制：

- 时间半径：`nms_time_radius`（GUI 输入，秒）
- 道半径：`nms_channel_radius = 1`（固定）

然后再次求最优轨迹，迭代直到无合法路径或达到 `max_tracks`。

### 3.6.1 NMS 的物理意义

同一辆车在邻近时间/邻近道上会对应同一簇峰。  
如果不抑制，DP 会在这簇峰附近反复提取“近乎同一条轨迹”的副本。

NMS 做的事是：  
每提取出一条轨迹，就在其周围挖掉一条“禁止带”，强迫下一次 DP 去找别的车。

### 3.6.2 NMS 过大/过小的后果

- `nms_time_radius` 太小：重复轨迹多
- `nms_time_radius` 太大：相邻车辆可能被误合并或漏掉

经验上：

- 车流密集：半径略减
- 重复提取明显：半径略增

## 3.7 后处理：断裂轨迹拼接（新）

为应对“轻微加减速 + 局部漏检”导致的碎段，新增拼接：

- 只拼接同方向轨迹
- 不能道号重叠
- 允许道间缺口 `dch <= max_gap_channels`（默认 8）
- 连接时间差 `dt` 需满足放宽后的速度窗（`dt_slack_ratio=0.35`）
- 连接速度和两段均速差不能过大（默认 `<=35 km/h`）

拼接代价：

`cost = gap_cost + 0.6*dt_cost + 0.8*speed_cost`

选择全局最小代价对逐步合并，直到无法再拼接。

这一步显著提高非匀速场景的连贯性。

### 3.7.1 拼接在数学上到底检查了什么

对两段轨迹 `left -> right`，核心检查：

1. 道号单调：`right` 必须在 `left` 的后续道上  
2. 时间-速度可行：`dt` 必须落在（放宽后的）速度窗  
3. 速度一致：连接速度和两段均速差不能太大

只有全部通过才允许进入“可拼接候选集”。

### 3.7.2 代价项拆解

`cost = gap_cost + 0.6*dt_cost + 0.8*speed_cost`

- `gap_cost`：跨道缺口越大，代价越高
- `dt_cost`：连接时间偏离速度窗中值越多，代价越高
- `speed_cost`：两段均速不一致越多，代价越高

所以拼接不是“最近就拼”，而是“同时满足空间、时间、速度一致性后，再选最低总代价”。

### 3.7.3 为什么要“迭代拼接直到不能拼”

真实情况常是：

- 轨迹被切成 3 段、4 段，而不是仅 2 段

因此需要每次拼完后重新评估新轨迹，继续找下一对可拼接片段，直到稳定。

## 4. 整小时模式与当前窗口模式

## 4.1 当前窗口模式

- 直接在 `data_view` 上提取
- 去重 -> 拼接 -> 再去重

## 4.2 整小时模式

- 分块提取：`tile_seconds`，重叠 `overlap_seconds`
- 跨块合并（按道号重叠和时间一致性）
- 全局去重
- 全局拼接
- 再去重

## 4.3 为什么要分块

1 小时、50 道、1kHz 的数据规模很大，若一次性建图：

- 峰节点量会非常大
- 内存和计算时间都会明显上升

分块的作用：

- 控制单次 DP 的规模
- 用 overlap 保证跨块边界不断轨
- 再通过 merge + dedup + stitch 恢复全局一致性

### 4.4 分块参数怎么选

- `tile_seconds`：越大越稳，越小越快
- `overlap_seconds`：至少覆盖“最大预期跨道传播时间 + 安全余量”

对 `dx=100m, vmin=60km/h`：

- 单道最大传播时间约 `6s`
- 实际建议 overlap 通常 `10~30s`

## 6. 参数选择的物理解释

1. **速度窗 `[vmin,vmax]` 本质是斜率先验**  
   速度越快，`dt` 越小；速度越慢，`dt` 越大。给定速度窗就是给轨迹斜率一个物理可行区间。

2. **`min_peak_distance` 与脉冲宽度匹配**  
   在 `fs=1000Hz` 下，`500` 样点约 `0.5s`，与高斯窗主瓣尺度同量级，可减少同一车在单道被重复取多个峰。

3. **`min_track_channels` 对应最小空间持续长度**  
   `12` 道、`dx=100m` 等价于至少 `1.2km` 的空间持续，能过滤随机点串联。

4. **速度变化容忍用于非匀速**  
   真实车辆存在微小加减速，若强行恒速惩罚会断轨。加入 `18 km/h` 容忍区后，可保持物理合理且更连续。

5. **拼接约束用于漏检恢复**  
   若中间若干道无可靠峰，拼接通过“道距-时间-速度一致性”恢复同车轨迹，避免把同一车拆成多段。

6. **模板增强开关对应“形态先验强度”**  
   开启时：假设车辆响应接近脉冲形，抗噪更强；  
   关闭时：直接用原始包络，保留细节但对噪声更敏感。

7. **`lambda_speed` 与 `v_tol` 决定“非匀速容忍度”**  
   - `v_tol` 大：允许更强加减速  
   - `lambda_speed` 小：速度突变惩罚更弱  
   二者共同决定轨迹是否容易“被硬拉直”或“被过度弯曲”。

8. **`lambda_skip` 与 `max_skip_channels` 决定“漏检容错力度”**  
   - `max_skip_channels` 大：可跨更多缺失道  
   - `lambda_skip` 大：虽可跳，但代价高（更谨慎）

## 7. 典型调参策略（按场景）

## 场景 A：轨迹断裂较多（你当前问题）

![image02.png](image02.png)

- 先保持：`speed_min/max` 覆盖真实速度
- 调小：`prominence`（如 `0.3~0.4`）
- 调小：`min_track_channels`（如 `8~10`）
- 调小：`min_peak_distance`（如 `350~450`）

作用：增加候选点密度，减少“断裂不可连”情况。

## 场景 B：误检串轨较多

- 调大：`prominence`
- 调大：`min_track_channels`
- 略调大：`nms_time_radius`

作用：收紧候选与轨迹质量门槛。

## 场景 C：速度波动明显（加速/减速）

- 适当加宽速度窗（例如 `50~140 km/h`）
- 保持当前增强版速度惩罚（已内置容忍区）

作用：允许轨迹斜率随时间缓慢变化。

## 场景 D：你说的“第三个点开始偏”

优先排查顺序：

1. 速度窗是否覆盖真实速度（尤其 `Speed min` 是否过高）  
2. `prominence` 是否过低（把噪声峰放进来）  
3. `min_peak_distance` 是否过小（同一道重复峰太多，DP易跳错）  
4. 模板增强是否更适合当前数据（开/关做 A/B 对比）  
5. 是否需要减小 `lambda_speed` 或增大 `v_tol`（允许更平滑转弯）

## 场景 E：轨迹“明明没峰也被连上”

这通常是“候选点太宽松 + 跳道容错 + 拼接”叠加导致：

- 先提高 `prominence`
- 再提高 `min_track_channels`
- 再略增 `nms_time_radius`
- 最后再考虑收紧速度窗

## 场景 F：希望偏保守（宁可漏一点也别串轨）

建议组合：

- `prominence` 上调
- `min_track_channels` 上调
- `speed_max` 适当下调
- 模板增强打开

## 场景 G：希望偏召回（宁可多抓再人工筛）

建议组合：

- `prominence` 下调
- `min_track_channels` 下调
- `max_skip_channels` 可增（代码内参数）
- 必要时关闭模板增强做对照

## 9. 已知局限与后续可升级方向

1. 当前仍是“峰点驱动”，若波形严重失真，峰检测质量会限制上限。  
2. 轨迹拼接是启发式贪心，可升级为全局最小代价匹配（Hungarian / min-cost flow）。  
3. 可加入“可选短缺口插值显示”（仅显示补点，不影响导出真峰点）。

## 10. 一句话总结

当前算法本质是：

**“物理速度约束下的峰点图动态规划 + 边界放宽 + 多段拼接”**，
在保留“只用真实峰点”的前提下，提高了非匀速与漏检场景下的轨迹连贯性。
